# 黄山派 Codex 宠物助手开发计划

更新时间：2026-07-15

## 目标

在现有黄山派 VibeBoard Runtime、BLE 语音链路和 GLM-ASR-2512 转录能力上，
实现一个可长期运行的 Codex 宠物助手：

- 板子唤醒后采集语音，由电脑转录并自动创建或继续 Codex 任务。
- 板子显示 Codex 的运行、等待输入、完成、阻塞和额度状态，并用 RGB 灯提醒。
- 电脑端 Codex 可通过受控 MCP 工具调用黄山派已开放的硬件能力。
- 后续接入外置喇叭、提示音、TTS，以及经过验证的唤醒词/VAD。
- API Key、Codex 登录凭据和敏感审批信息只保留在电脑端。

## 总体架构

```text
黄山派 codex_pet App
  <-> BLE voice + pet/v1
电脑 codex_pet_bridge（唯一 BLE owner）
  -> GLM-ASR-2512
  <-> Codex app-server
  <- Codex hooks
  <- Codex quota provider
  <-> Huangshan MCP server
```

Bridge 必须统一串行化语音、App 管理、状态、额度和 MCP 硬件命令，不能让多个
进程同时抢占同一条 BLE 连接。

## 阶段门禁

每个阶段完成后必须连续进行三轮检查。任何一轮发现问题，都必须先修复并重新从
受影响的检查轮次开始，三轮全部通过后才能进入下一阶段。

1. 第一轮：协议、静态检查、格式、离线单元测试和契约一致性。
2. 第二轮：模块集成、模拟传输、异常路径、超时、重复事件和断线恢复。
3. 第三轮：全仓回归、构建、真机链路或明确记录无法执行的硬件门禁。

检查记录必须包含命令、结果、发现的问题和修复提交/文件，不能只写“通过”。

### 阶段 3：状态提醒、额度与安全可靠性（已完成）

交付物：

- 接入 Codex hooks 和 app-server 流式事件，聚合多任务状态。
- 接入 5 小时/周额度、重置时间、数据来源、更新时间和 stale/error 状态。
- 状态语义采用 `Running / Needs input / Ready / Blocked`，增加本地语音状态。
- RGB 灯支持颜色、闪烁模式、优先级、TTL 和断线自动清理。
- 加入 BLE 设备绑定与加密；未加密连接不得承载审批、凭据提示或任务正文。

推荐灯效：

- listening/transcribing：青色常亮或呼吸。
- running：蓝色缓慢呼吸。
- needs input：黄色快速闪烁，屏幕显示 approval/question/credential 子类型。
- ready：绿色双闪。
- blocked/error：红色快速闪烁。
- disconnected：熄灭或白色慢闪。

验收标准：

- 状态变化到板端显示的本地链路延迟目标小于 700 ms。
- quota 不可用时显示 stale/unavailable，不能错误显示为 0%。
- 断线、Bridge 崩溃和 Codex 重启不会遗留永久灯效。

### 阶段 4：MCP 双向硬件调用与实体审批（已完成）

交付物：

- 新增 Huangshan MCP server，通过 Bridge 调用 `RuntimeTransport`。
